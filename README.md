# Agentic Debt Research Pipeline — RQ3

This pipeline analyses technical debt (code smells) introduced by multi-agent software systems (MAS), and evaluates whether a dedicated refactoring agent can mitigate it. It uses two smell detection tools — **PyExamine** and **DPy** — across per-turn snapshots and three independent refactoring runs (v1, v2, v3).

---

## Repository Layout

```
python_smells_detector/
├── pkls/                        # Input: one .pkl file per project
├── DPy/                         # DPy binary directory
│   ├── DPy                      # The DPy executable
│   └── output/                  # DPy writes JSONs here
├── refactored_code/             # Refactored output v1 (already exists)
├── refactored_code_v2/          # Refactored output v2 (generated by refactor_agent.py)
├── refactored_code_v3/          # Refactored output v3 (generated by refactor_agent.py)
├── rq3_workdir/                 # Working directory: per-turn snapshots and reports
├── code_quality_config.yaml     # PyExamine configuration
├── rq3_pipeline.py              # Main smell analysis pipeline
├── refactor_agent.py            # Refactoring agent (produces v2 and v3)
├── rq3_analysis_v3.ipynb        # Analysis notebook (RQ1 + RQ3)
├── .env                         # API key (not committed)
└── venv/                        # Python virtual environment
```

## Prerequisites

### 1. Python virtual environment

The virtual environment must be active whenever you run the pipeline or the refactoring agent, because PyExamine's `analyze_code_quality` command is installed inside it.

```bash
source venv/bin/activate
```

### 2. Environment variables

Create a `.env` file in the project root:

```
ANTHROPIC_API_KEY=your_key_here
```

### 3. DPy binary

The `DPy` binary must be present at `./DPy/DPy` and be executable:

```bash
chmod +x ./DPy/DPy
```

## Step 1 — Generate refactored code (v2 and v3)

> Skip this step if `refactored_code_v2/` and `refactored_code_v3/` already exist and are fully populated.

`refactored_code/` (v1) is assumed to already exist. The refactoring agent will produce v2 and v3 independently, both reading from the same original last-turn source in `rq3_workdir/`.

```bash
source venv/bin/activate
python refactor_agent.py
```

This will:
- Read the last turn of each project from `rq3_workdir/`
- Call Claude (claude-haiku-4-5) once per file per version
- Write output to `refactored_code_v2/` and `refactored_code_v3/`
- Skip any project that already has output (safe to re-run if interrupted)

**Expected runtime:** depends on number of projects and files, plus any rate limit waits. Typically 20–60 minutes for 30 projects.

---

## Step 2 — Run the smell analysis pipeline

This runs both PyExamine and DPy on every per-turn snapshot and on all three refactored versions.

```bash
source venv/bin/activate
python rq3_pipeline.py \
    --pkl_dir        ./pkls \
    --config         ./code_quality_config.yaml \
    --dpy_dir        ./DPy \
    --output         rq3_results.csv \
    --work_dir       rq3_workdir \
    --refactored_dirs ./refactored_code ./refactored_code_v2 ./refactored_code_v3
```

### Arguments

| Argument | Required | Description |
|---|---|---|
| `--pkl_dir` | Yes | Directory containing `.pkl` project files |
| `--config` | Yes | Path to `code_quality_config.yaml` |
| `--dpy_dir` | No | Path to the `DPy/` folder containing the binary. Omit to skip DPy analysis |
| `--output` | No | Output CSV path (default: `rq3_results.csv`) |
| `--work_dir` | No | Working directory for snapshots and reports (default: `rq3_workdir`) |
| `--refactored_dirs` | No | One or more refactored code root directories, space-separated. Auto-labelled v1, v2, v3 in order |

### Output files

| File | Description |
|---|---|
| `rq3_results.csv` | One row per turn. Contains per-turn smell counts from both tools, plus v1/v2/v3 refactored counts and deltas |
| `rq3_results_smells_detail.csv` | One row per smell instance. Used for smell-name breakdowns in the notebook |

### Caching behaviour

The pipeline caches all intermediate results in `rq3_workdir/`. If you re-run it:
- PyExamine is skipped for any turn whose report CSV already exists
- DPy is skipped for any turn whose `_implementation_smells.json` already exists
- The same applies to refactored snapshots

This means re-runs are fast and safe after partial failures.

## Output column reference

### Per-turn columns (PyExamine)

| Column | Description |
|---|---|
| `structural_smells` | High cyclomatic complexity, too many branches, etc. |
| `code_smells` | Feature envy, divergent change, temporary field, etc. |
| `architectural_smells` | Improper API usage, orphan modules, etc. |
| `total_smells` | Sum of the above |

### Per-turn columns (DPy)

| Column | Description |
|---|---|
| `dpy_implementation_smells` | Long statements, complex methods, magic numbers, empty catch blocks |
| `dpy_design_smells` | Broken modularisation, multifaceted abstraction |
| `dpy_architecture_smells` | Architecture-level issues (often zero) |
| `dpy_total_smells` | Sum of the above |

### Refactored version columns

For each version `vN` (v1, v2, v3) and each tool, the following columns are added:

| Column | Description |
|---|---|
| `vN_refactored_total_smells` | Total smells in the refactored snapshot |
| `vN_delta_ref_vs_baseline_total` | Refactored minus baseline (negative = improvement) |
| `vN_delta_ref_vs_postreview_total` | Refactored minus post-review (negative = improvement) |

The same pattern applies for `_structural_`, `_code_`, `_architectural_`, and `_dpy_*` variants.

## Troubleshooting

**`analyze_code_quality` not found**
The venv is not active. Run `source venv/bin/activate` first.

**`NotADirectoryError: [Errno 20] Not a directory: './DPy/DPy'`**
Pass the folder containing the binary, not the binary itself. Use `--dpy_dir ./DPy`, not `--dpy_dir ./DPy/DPy`.

**DPy parse error in logs (SyntaxError)**
DPy logs parse errors for files with syntax issues and continues. These are warnings, not failures — the affected file is skipped and other files in the project are still analysed.

**Rate limit errors during refactoring**
The refactoring agent has built-in retry logic with exponential backoff. It will wait and resume automatically. If it is interrupted, re-running the script will skip already-completed projects.

**Missing refactored columns in CSV**
Ensure you pass all three directories to `--refactored_dirs` and that each directory contains at least one `.py` file per project. Projects with no `.py` files are skipped and their refactored columns will be empty.
